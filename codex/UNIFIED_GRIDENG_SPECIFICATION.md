# GridEng — единая спецификация требований для Codex

Дата ревизии: 2026-05-16  
Источник: комплект `codex.zip`, проектный контекст GridEng и ранее согласованные решения по веткам `cross_sections`, `materials`, `UI revision 2`, `UI revision 3`, `UI revision 3.1`.

## 1. Назначение документа

Документ сводит разрозненные требования к GridEng в единый источник для дальнейшей постановки задач Codex.

Спецификация не заменяет исходные рабочие документы, но задает приоритет при противоречиях:

1. `codex/IMPLEMENTATION_PLAN.md` — порядок выполнения и статус задач.
2. `codex/UNIFIED_GRIDENG_SPECIFICATION.md` — консолидированная целевая архитектура и снятые противоречия.
3. `codex/reference/*_CHECKED_REQUIREMENTS.md` — исходные проверенные UI-требования с комментариями.
4. `codex/IMPLEMENTATION_CROSS_SECTIONS_STRUCTURE.md` и `codex/IMPLEMENTATION_MATERIALS_STRUCTURE.md` — backend-онтология каталогов профилей и материалов.
5. `codex/docs/*.md` — технические заметки по конкретным итерациям.

## 2. Архитектурные инварианты

- Проект реализуется малыми атомарными задачами с ограниченным diff.
- `backend/app/lib` и расчетное ядро не изменяются без отдельного прямого решения.
- Полноценная SQL-БД на текущем этапе не вводится; нормативные данные профилей и сталей хранятся в JSON.
- Frontend работает на React + TypeScript + Vite + MUI + Three/R3F + Zustand.
- Backend работает на FastAPI; текущие API не версионируются через `/api/v1` в рамках этих задач.
- Все frontend-задачи завершаются `cd frontend && npm run build`; задачи, меняющие frontend-код, дополнительно завершаются `cd frontend && npm run lint`.
- Все пользовательские команды меню, hotkeys и консоли должны использовать единый `command registry`.
- Все ошибки, предупреждения, служебные сообщения и placeholders показываются через единый MUI `Snackbar` + `Alert` снизу по центру.
- Локализация пользовательского UI централизована, минимальные языки — `ru` и `en`, переключение языка только через консоль.

## 4. GridEngModel v0.2

### 4.1. Координаты и единицы

- Геометрия модели хранится в 3D.
- Нагрузки и закрепления задаются в ГСК.
- Локальный базис стержня вычисляется отдельно и не подменяет исходную модель.
- Ветровое направление хранится как 3D-вектор, но UI текущей итерации редактирует только `X/Y`; при сохранении `direction.z = 0`.
- Давление ветра хранится и вводится в `Па`; отрицательное давление запрещено, `0` допустим.

### 4.2. Профиль, материал и элемент

- `Member` остается расчетной сущностью элемента стержневой модели.
- Профиль хранит геометрию/сечение.
- Материал остается свойством элемента (`member.materialId` или текущий эквивалент в модели), а не свойством профиля.
- Допустимое будущее расширение: `defaultMaterialId` у профиля как значение по умолчанию при создании элемента, но оно не заменяет материал элемента.
- Выбор стали выполняется после выбора профиля: сначала выбран профиль, затем по расчетной толщине профиля и типу проката фильтруются материалы.

### 4.3. Нагрузки

- Текущая версия модели — `GridEngModel v0.2`.
- `Load` разделяется на `kind: 'force' | 'moment'`.
- Одна пользовательская нагрузка не должна одновременно содержать силу и момент; при миграции из старого формата force/moment разделяются на две нагрузки.
- Поддерживаются узловые сосредоточенные нагрузки и линейные распределенные нагрузки по стержню.
- `function`-distribution зарезервирована для будущего, но не имеет редактора и не участвует в расчете/визуализации в текущей итерации.

### 4.4. Закрепления

- Закрепления отображаются и редактируются по 6 DOF.
- Визуальный язык закреплений должен показывать ограничения по осям, но не менять расчетную геометрию.

## 5. DXF import v0.1 / UI revision 3.1

### 5.1. Граница DXF-импорта

- Поддерживается только импорт DXF `LINE`.
- Широкий DXF-импорт других entities не выполняется.
- Игнорируемые entities должны фиксироваться в логах/diagnostics, а не молча теряться.
- DXF preview не пишет в `modelStore`, пока пользователь не нажал `Импорт`.

### 5.2. Preview и диагностика

- Preview нормализуется отдельно от основной модели.
- После нормализации preview: `Z >= 0`, модель центрирована относительно оси `Z`.
- Повороты preview вокруг осей и `reset` применяются к preview-состоянию, не к основной модели.
- Режимы окраски:
  - `Диагностика` — зеленый/желтый/красный по structured diagnostics;
  - `Профили` — цвет назначенного каталогового профиля либо исходный DXF color/color group, если профиль еще не назначен.

### 5.3. Вкладки DXF import dialog

Текущая целевая структура:

```text
Шапка → Tabs: Модель | Профили | Логи → нижний блок icon-кнопок
```

- Вкладка `Модель`: 3D preview занимает доступное пространство.
- Вкладка `Профили`: только таблица `Цвет | Тип профиля | Стандарт | Профиль`.
- Вкладка `Логи`: компактные агрегированные сообщения без raw JSON в основном UI.
- Вся служебная информация, счетчики, debug и JSON/details переносятся в `Логи`.

## 6. Backend: cross_sections

### 6.1. Назначение

Цепочка блока:

```text
Нормативная JSON-БД сортамента → data-layer → Crossection factory → FastAPI API → frontend catalog/dataframe
```

### 6.2. Принцип хранения

- Продуктивная JSON-БД сортамента хранит только нормативную геометрию и источник ГОСТ.
- Расчетные характеристики не хранятся в БД и вычисляются через `Crossection`/API.
- Один JSON-файл описывает один стандарт.
- `standard`, `source`, `validation` хранятся один раз в корне файла, а не в каждом профиле.

### 6.3. Запрещенные поля в продуктивном JSON

```text
standards
confidence
reference_values
source внутри элемента
standard внутри элемента
validation внутри элемента
steel_group
```

### 6.4. API

Текущий префикс:

```text
/api/cross-sections
```

Frontend должен использовать API-контракт профилей как основной источник справочника; локальный каталог допустим только как fallback или временная заглушка до интеграции.

## 7. Backend: materials

### 7.1. Назначение

Цепочка блока:

```text
Нормативная JSON-БД сталей → data-layer → resolver Material → FastAPI router → frontend material selector
```

### 7.2. Принцип хранения

- Продуктивная JSON-БД сталей хранит только расчетные параметры материала и нормативные источники.
- Геометрия профилей хранится отдельно в `cross_sections`.
- Один материал может иметь несколько нормативных источников и несколько диапазонов свойств по толщине.
- Не допускается создавать отдельные материалы-дубли только из-за разных ГОСТов, если это одна и та же марка стали в онтологии проекта.
- `key` пишется латиницей, `display_name` — кириллицей по ГОСТ.

### 7.3. Расчетные параметры

```text
Rt    — расчетный/нормативный предел текучести, MPa
Rb    — временное сопротивление, MPa
E     — модуль упругости, MPa
G     — модуль сдвига, MPa
alpha — коэффициент линейного расширения, 1/C
rho   — плотность, kg/m3
```

Химический состав, ударная вязкость, углеродный эквивалент, категории поставки и состояние поставки не включаются в текущую БД.

### 7.4. API

Текущий префикс:

```text
/api/materials/steels/...
```

Frontend material selector должен вызывать backend resolver/endpoint с учетом выбранного профиля, расчетной толщины и типа проката.

## 8. UI: дерево проекта, каталоги и свойства

- В дереве проекта остаются расчетные сущности: `Нагрузки`, `Элементы`, `Ноды`.
- `Профили` и `Материалы` не должны возвращаться в дерево проекта как раскрываемые разделы.
- Справочники профилей и материалов открываются через отдельные диалоги/каталоги.
- Каталоги проекта и глобальные справочники — разные понятия:
  - глобальный справочник берет данные из backend JSON/API;
  - каталог проекта показывает только используемые в модели профили/материалы, если такой экран потребуется.
- Properties panel редактирует выбранную расчетную сущность; для `Member` показывает профиль и материал как свойства элемента.

## 9. UI: тема, меню, команды, консоль

- Базовая эстетика — black & white / grayscale.
- Цвет остается функциональным инженерным слоем: диагностика, нагрузки, моменты, закрепления, profile color, selected state, alerts.
- `shape.borderRadius = 0` является целевым решением для строгого интерфейса, если это не ухудшает читаемость.
- Все icon buttons имеют tooltip и `aria-label`.
- Placeholder-иконка `NewReleasesIcon` допускается только с явным TODO/tooltip, чтобы она не стала финальной без решения.
- Опасные команды требуют подтверждения или undo-механики.
- Консоль — нижняя docked-панель + fullscreen modal; она использует тот же command registry, что меню и hotkeys.

## 10. Снятые противоречия

| Противоречие / риск | Итоговое решение |
|---|---|
| `Профили` и `Материалы` одновременно предлагались как разделы дерева и отдельные каталоги | В дереве остаются `Нагрузки`, `Элементы`, `Ноды`; профили и материалы открываются отдельными диалогами/каталогами |
| Материал предлагался как свойство профиля | Материал остается свойством элемента; профиль — геометрия/сечение; `defaultMaterialId` возможен позже только как default |
| Одна нагрузка могла содержать силу и момент одновременно | В v0.2 нагрузка имеет `kind`; force/moment разделяются на отдельные нагрузки |
| `Открыть`/`Сохранить` могли дублировать JSON import/export | Для локального файла это aliases `Импорт JSON` / `Экспорт JSON`; серверное хранение остается placeholder до отдельной задачи |
| Цветная диагностика конфликтует с grayscale-темой | Grayscale применяется к UI; инженерные цвета допускаются как смысловой слой сцены/диагностики |
| DXF preview мог менять основную модель до импорта | Preview изолирован; запись в `modelStore` только после явного `Импорт` |
| Ветровой вектор 3D, но UI требует 2D | Модель остается 3D; UI редактирует X/Y и сохраняет `z=0` |
| Давление ветра встречалось в кПа/Па | Для текущего UI и модели зафиксировано `Па` |

## 11. Открытые вопросы, которые не блокируют текущие задачи

1. Будущий нормативный расчет ветра: методика, СП/нормативная база, параметры местности и высоты — отдельная итерация после ручного ввода.
2. Серверное хранение проектов: пока placeholder, так как текущие задачи используют JSON import/export.
3. `defaultMaterialId` у профиля: допустимо только как будущий default, не как замена `member.materialId`.
4. Расширение DXF beyond `LINE`: не входит в текущую спецификацию.
5. SQL-БД для нормативных справочников: не вводится до отдельного решения.
