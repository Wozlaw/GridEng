# Task 21.3 — Active load case: единое активное загружение и дерево проекта

## Цель

Добавить единое состояние `activeLoadCaseId`, чтобы дерево проекта, сцена, ветер, карта напряжений и редакторы работали с одним и тем же загружением.

## Что сделать

1. В `modelStore` или существующий UI-store добавить:
   - `activeLoadCaseId: string | null`;
   - `setActiveLoadCaseId(id: string | null)`;
   - selector для получения активного `LoadCase`.
2. Fallback-логика:
   - при загрузке/импорте модели выбрать первое загружение, если активное отсутствует;
   - при удалении/замене модели сбрасывать active id на существующее первое загружение или `null`;
   - не хранить id, которого нет в `model.loadCases`.
3. В `ProjectTreePanel` в разделе `Нагрузки` заменить плоский список на выпадающий выбор существующего загружения.
4. При смене загружения менять список нагрузок в дереве.
5. Из элементов нагрузок убрать номиналы: оставить только наименования усилий/моментов/распределенных нагрузок.
6. Подготовить использование `activeLoadCaseId` для следующих задач:
   - `Viewport3D`;
   - `LoadVectors`;
   - `MomentVectors`;
   - `StressMapLegend`;
   - `WindEditorDialog`;
   - `PropertiesPanel`, если там показывается контекст нагрузок.

## Ограничения

- Не менять схему `GridEngModel`, если `activeLoadCaseId` относится только к UI-состоянию.
- Не записывать `activeLoadCaseId` в экспорт JSON модели, если это не предусмотрено текущей модельной спецификацией.
- Не реализовывать расчет сочетаний нагрузок.

## Ожидаемые места изменений

- `frontend/src/app/store/*`
- `frontend/src/widgets/project-tree/*`
- `frontend/src/widgets/viewport-3d/*`
- `frontend/src/widgets/properties-panel/*` при необходимости

## Критерии готовности

- Переключение загружения в дереве меняет список нагрузок.
- Состояние активного загружения доступно сцене и редактору ветра.
- При импорте модели нет битых ссылок на несуществующее загружение.
- JSON export модели не загрязняется UI-only состоянием.

