# Task 22.3 — DXF preview diagnostics data model

## Цель

Расширить DXF preview-данные так, чтобы 3D preview мог подсвечивать не всю модель целиком, а конкретные элементы/группы по статусу импорта.

## Требования

1. Сохранить существующую онтологию DXF preview и модели; не менять `GridEngModel` без необходимости.
2. Расширить типы DXF preview локальными диагностическими структурами, например:
   - `memberImportStatuses` / `memberDiagnostics` с привязкой к временным `memberId` или DXF handle/layer/color group;
   - `nodeImportStatuses` / `nodeDiagnostics`, если уже есть стабильная привязка узлов;
   - `colorGroups` / `profileGroups` для связи исходного DXF color → группа элементов.
3. Статусы минимум:
   - `ok` — зеленый;
   - `warning` — желтый;
   - `error` — красный.
4. Для каждой диагностики хранить машиночитаемый `code` и пользовательское `message`.
5. Не считать достаточным решение “окрасить всю preview-модель по общему статусу”. Нужна привязка хотя бы к стержням/группам.
6. В диагностику включить минимум:
   - нулевые/ошибочные линии;
   - висячие стержни, если определение уже есть;
   - группы с неназначенным профилем;
   - ignored entities в логах.
7. Существующие warnings/errors строками можно оставить для обратной совместимости, но новые UI-компоненты должны использовать структурированные diagnostics.

## Ограничения

- Не применять preview к `modelStore` до нажатия “Импорт”.
- Не переписывать весь DXF-парсер.
- Не подключать backend/API.

## Критерии готовности

- Типы DXF preview позволяют определить цвет конкретного элемента/группы в режиме диагностики.
- Логи импорта могут показать summary и структурированные diagnostics.
- Старые сценарии DXF-импорта не сломаны.
- Build/lint проходят.

## Ручная проверка

1. Импортировать простой DXF без ошибок.
2. Импортировать DXF с неназначенными профилями/цветами.
3. Проверить, что diagnostics доступны в preview/log data и не применены к основной модели до подтверждения импорта.

cd frontend
npm run build
npm run lint
