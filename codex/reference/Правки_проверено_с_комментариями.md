# 1. Общие замечания к интерфейсу

### 1.1 Цветовое оформление

- Цветовую палитру оформить в стиле black & white minimalism с переключением между светлой и темной темой, цвет используем в оформлении смысловых акцентов

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Цветовые акценты должны быть вынесены в единые токены темы MUI: `success`, `warning`, `error`, `info`, `selected`, `profileColor`, `restraintColor`, `loadForceColor`, `loadMomentColor`. Не задавать цвета хаотично внутри компонентов.

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> Профильные цвета элементов уже используются как смысловая дифференциация профилей. Их нужно оставить как инженерный слой сцены, даже если общая UI-палитра black & white.

- Все должно быть на русском языке

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Вынести пользовательские подписи в единый словарь/константы локализации. Сейчас в интерфейсе и типах режимов просмотра присутствуют английские строки (`Wireframe`, `Real`, `Loads`, `Restraints`, `Stress map`, `Model valid`, `DXF tolerance` и т.п.), их нужно заменить централизованно, а не точечно.

- Области дерева проекта (ProjectTree) и свойств (Properties) должны быть изменяемыми по ширине (должны быть заданы минимальная и максимальная ширина) и иметь возможность сворачивания (скрытия, кнопка сворачивания по центру границы со сценой)

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Состояние ширины/свернутости панелей желательно сохранять в `localStorage`, чтобы после перезагрузки рабочее пространство не сбрасывалось. Минимумы/максимумы задать явно, например: ProjectTree `240–420 px`, Properties `280–520 px`, свернутое состояние `0 px` + видимая кнопка восстановления.

- В оформлении кнопок, везде где это возможно, необходимо использовать иконки. Если иконку не удается подобрать по смыслу (подходящий вариант отсутствует в mui библиотеке), то необходимо использовать "NewReleasesIcon" с окраской в #ff9800. Для каждой кнопки должен быть определен tooltip

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> `NewReleasesIcon` лучше считать временным placeholder-иконкой. В tooltip или комментарии к конфигурации кнопки нужно оставлять `TODO: подобрать финальную иконку`, иначе placeholder может остаться в продуктивном интерфейсе.

- При оформлении Icon Button использовать цветовые смысловые акценты

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Цвет не должен быть единственным носителем смысла. Для критичных команд обязательно: tooltip + aria-label + понятная иконка. Для опасных команд (`сброс`, `удалить`, `очистить`) использовать подтверждение либо undo-механику.

- Оформление сцены и меню без скругленных углов, более строгое 

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> В теме MUI установить `shape.borderRadius = 0` и переопределить `Paper`, `AppBar`, `Button`, `ToggleButton`, `Menu`, `Dialog` только там, где это не ухудшает читаемость. Для диалогов допустимо оставить минимальное скругление только если будет принято отдельным UI-решением.

## 1.2 Меню (TopMenu)

- Меню реализуем по принципу ленты (как в современных CAD приложениях, например fusion 360)

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Реализацию лучше делать через конфигурационную модель меню: массив блоков → массив команд → привязка к handler/consoleCommand/hotkey/tooltip/icon. Это сразу позволит переиспользовать те же команды в консоли.

- Слева (в начале) логотип (пока вставь нейтральный исключительно как placeholder)

- После логотипа у нас идут последовательно блоки меню

- Блок меню состоит из плитки с кнопками (icon button). Расположение кнопок слева на право по мере уменьшения частоты использования. Кнопки могут группироваться тремя способами (группировка по смыслу):
  
  - Одна кнопка  64 px
  
  - две (вертикально), либо четыре кнопки (плитка) 32 px
  
  - три (вертикально), либо шесть (плитка В х Г -> 3 х 2), либо девять (плитка 3 х 3) кнопок 16 px

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> Кнопка 16 px как интерактивная зона слишком мала для нормального попадания мышью. Предлагаю трактовать `16/32/64 px` как размер иконки/визуальной плотности, но не как полный hitbox. Минимальная кликабельная зона для мелких команд должна быть не меньше `28–32 px`, иначе интерфейс станет неудобным в CAD-сценарии.

- Под кнопками блока меню выпадающий список с функционалом оформленным как  иконка - команда. Титул выпатающего списка - наименование блока. Если в блоке нет дополнительных команд (выпадающий список пустой), то Титул выводится как текстовое поле.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Для каждого блока нужен единый источник команд: часть команд выводится плитками, остальные — в выпадающем списке. Команда не должна дублироваться одновременно в плитке и dropdown без причины.

- Блок меню должен быть организован эргономично с прицелом на пользователя

- Блоки меню слева на право: Компонент (импорт/экспорт/открыть/сохранить) -> Режим просмотра -> Видимость ->  Каталог (конструкции/профили/материалы) -> Аналитика (добавить кнопку Консоль, добавить кнопку определения ветровой нагрузки) -> Отчет (меню отчетов: спецификация, отчет по расчету) -> О программе (справка/лицензия и пр.)

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> `Конструкции` в блоке `Каталог` сейчас не имеют отдельной сущности в модели. Нужно определить: это каталог сохраненных моделей/секций/типовых конструкций или будущая библиотека сборок. До определения лучше оставить пункт как disabled/placeholder.

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> `Определение ветровой нагрузки` сейчас не имеет расчетного алгоритма: в модели есть только `wind.direction` и `wind.nominalPressureKPa`. Нужно явно разделить: на этом этапе делаем диалог ручного задания ветра или уже калькулятор по нормативу. Для калькулятора нужны исходные параметры и нормативная методика.

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> `Отчет по расчету` возможен только после появления расчетных результатов (`results`) или backend-расчета. Пока логичнее сделать пункт disabled/placeholder, а активной оставить только будущую `Спецификацию` по текущей модели.

- Всю служебную информацию с меню убираем 

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Уточнить правило: убираем debug/dev информацию (`validation errors`, `model id`, `DXF tolerance`, raw ids), но оставляем пользовательски полезные статусы в status bar и/или диалогах.

## 1.3 Status bar

- Статус (левый край) показываем текстовой строкой по шаблону: Наименование проекта/Количество нод/ Количество элементов/ ... то есть меняем чипы на приведенную структуру, все счетчики выносим в статусную строку

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Рекомендуемый шаблон: `Проект: {name} / Ноды: {n} / Элементы: {m} / Нагрузки: {l} / Закрепления: {r} / Профили: {p} / Материалы: {mat} / Статус: {valid|ошибки|предупреждения}`. Выбранный объект можно показывать справа: `Выбрано: ...`, но без raw debug-полей.

- Справа убираем "DXF tolerance"

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> `DXF tolerance` не удалять из состояния проекта/импорта, а перенести в настройки импорта DXF или настройки проекта.

## 1.4 Сцена

- Гизмо переносим в правый верхний угол

- Всю служебную информацию убираем

- Кнопку Fit переносим с меню (Режим просмотра), определяем на комманду горячие клавиши

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> По смыслу должно быть: `кнопку Fit переносим В меню, блок Режим просмотра`. Сейчас Fit расположен поверх сцены. Команду также нужно зарегистрировать в консоли и горячих клавишах, например `fit`, `view.fit`, `F` или `Ctrl+F` — комбинацию лучше выбрать отдельно, чтобы не конфликтовать с поиском браузера.

- Оси и сетка - добавить возможность отключения. Кнопки в меню (Режим просмотра)

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> Оси и сетка логически относятся к блоку `Видимость`, а не к `Режим просмотра`. Можно оставить в `Режим просмотра`, но тогда блок `Видимость` станет менее последовательным. Рекомендация: оси/сетка — в `Видимость`, Fit/проекции/режимы камеры — в `Режим просмотра`.

- Элементы сделать цилиндрами, сейчас слишком тонкие не удобно попадать при необходимости выделения

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> Нельзя подменять расчетную геометрию цилиндрами. Нужно сделать цилиндры/утолщенные линии только как визуальный и picking-слой. Радиус должен быть экранно-зависимым или масштабируемым от габарита модели, иначе мелкие элементы станут слишком толстыми, а крупные — всё равно неудобными.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Для режима `Каркас` использовать `Line2`/цилиндрический proxy для выбора; для режима `Реальный вид` в дальнейшем использовать контур/геометрию профиля. Это не одно и то же.

- Выделение элементов сделать более акцентированным

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Для выбранного элемента использовать одновременно: увеличение толщины, контрастный цвет/обводку, небольшой glow/halo, приоритет рендера поверх невыбранных элементов. Не полагаться только на цвет.

- при выборе (выделении элементов) можель не должна менять своей позиции (масштаба/поворота) на сцене - должна оставаться статичной

  <span style="color:#1565c0; font-weight:700">КОНТЕКСТ:</span> Это правильное требование. Selection не должен вызывать `fitToModel`, `camera.lookAt`, изменение `OrbitControls.target` или пересчет позиции камеры. Fit должен выполняться только по отдельной команде пользователя.

- При выделении ноды с защемлением, защемление тоже подсвечивать на сцене

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Так как закрепления хранятся отдельной сущностью `restraints[]`, подсветку нужно делать по связи `restraint.nodeId === selectedNode.id`, а не ожидать поле `node.restraint` внутри ноды.

- Отображать дифференциацию защемлений по осям

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Нужно определить визуальный язык для 6 степеней свободы: `UX/UY/UZ` — поступательные закрепления, `RX/RY/RZ` — вращательные. Рекомендация: отдельные маркеры/штрихи по осям + tooltip с расшифровкой, а не один общий значок `заделка`.

## 1.5 Дерево проекта

- Заголовок по центру меню, снизу отделить чертой

- Состав проекта - более уместное наименование

- Структуру модели убираем - это информация присутствует в Status bar (дублировать не надо)

- Блоки Профили. Материалы, Нагрузки, Элементы, Ноды сделать сворачиваемыми

- Профили - убираем. Список используемых в проекте профилей должен быть доступен через кнопку меню (блок Аналитика)

- Материалы - убираем, материалы привязаны к профилю через свойства. Список используемых в проекте материалов должен быть доступен через кнопку меню (блок Аналитика)

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> Здесь есть внутреннее противоречие: сначала указано сделать блоки `Профили` и `Материалы` сворачиваемыми, затем — убрать их из дерева. Итоговое решение нужно зафиксировать так: в дереве остаются только `Нагрузки`, `Элементы`, `Ноды`; `Профили` и `Материалы` убираются из дерева и открываются через отдельные диалоги/каталоги.

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> Фраза `материалы привязаны к профилю через свойства` противоречит текущей модели и следующему пункту про элемент. Сейчас `materialId` хранится у элемента (`Member`), а профиль хранит геометрию/сечение. Это инженерно правильнее: один и тот же профиль может быть из разных сталей. Предлагаемое решение: материал остается свойством элемента; у профиля можно позже добавить `defaultMaterialId`, но не заменять `member.materialId`.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Кнопку списка используемых профилей/материалов лучше разместить в `Каталог`, а в `Аналитика` оставить инженерные сводки: масса по профилям, длины по профилям, материалы по элементам, нагрузки, ветер, проверки.

- Нагрузки - должен быть список нагрузок, при нажатии на элемент должно происходить его выделение и выделение соответствующей нагрузки на сцене, а в меню свойств отображаться свойство нагрузки

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> В текущем frontend selection поддерживает `node/member/profile/material/loadCase`, но не поддерживает отдельную `load`. Для этого требования нужно добавить выбор сущности `load`, например `SelectedEntity = { type: 'load'; id: load.id; loadCaseId: loadCase.id }` либо отдельный индекс `activeLoadCaseId + selectedLoadId`.

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> В дереве `Нагрузки` нужно показывать не только load case, а структуру `Расчетный случай → нагрузки`. Иначе непонятно, к какому расчетному случаю относится выбранная нагрузка.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> При выборе нагрузки на сцене подсвечивать: вектор нагрузки, точку приложения, связанную ноду/элемент. В Properties показывать именно конкретную нагрузку, а не весь load case.

- Элементы - Для восприятия удобнее писать не id ноды, а наименование профиля этой ноды (наименование выделять цветом в соответствии с цветовой диференциацией элементов на сцене. либо перед наименованием иконка соответствующего цвета, показывающая характер профиля). наименование смежных ноды можно не отображать

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> Здесь, вероятно, имеется в виду `наименование профиля элемента`, а не `профиля ноды`. Нода не имеет профиля. Предлагаемая строка элемента: `{label элемента или M-номер} — {profile.name}` + цветовой маркер профиля. При наведении tooltip может показывать `startNode.label → endNode.label`.

- Ноды - убрать служебную информацию, оставляем только Label 

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Если `node.label` отсутствует, показывать человекочитаемый fallback: `Нода {порядковый номер}` или `Нода {id}` только в tooltip. Иначе часть нод станет визуально пустой.

## 1.6 Меню свойств

- Заголовок по центру меню, снизу отделить чертой

- Свойства профилей и материалов - убираем, просмотр через отдельный диалог, вызываем по кнопке меню

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> В свойствах элемента оставить краткое отображение назначенного профиля/материала с кнопками `Выбрать/заменить` и `Открыть карточку`, но полную карточку профиля/материала показывать в отдельном диалоге.

- Добавить свойства нагрузок, на каждую выбранную нагрузку отдельное отображение. Редактируемые параметры: Координаты еденичного вектора, модуль нагрузки, нода приложения нагрузки, тип нагрузки (сила/момент - выпадающий список), Наименование (отображается в составе проекта)

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> Текущая модель нагрузки хранит `vector.force` и `vector.moment` одновременно, без поля `type`. Требование `тип нагрузки: сила/момент` означает изменение модели или UI-адаптер. Нужно решить:  
  1) добавить в модель `load.kind: 'force' | 'moment'`;  
  2) либо оставить модель как есть, а в UI считать нагрузку `сила`, если `force != 0`, и `момент`, если `moment != 0`.  
  Я рекомендую вариант 1 для пользовательского редактирования, но с конвертацией старой структуры.

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> Поле `нода приложения нагрузки` покрывает только узловые нагрузки. Текущая модель допускает `target: node | member`. Если нагрузки на элемент пока не нужны — явно ограничить UI узловыми нагрузками. Если нужны — в Properties должен быть `тип цели: нода/элемент` + соответствующий выбор.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> При вводе `единичный вектор + модуль` сохранять в модель компоненты: `force = direction * magnitude` или `moment = direction * magnitude`. Нужно нормализовать направление, проверять нулевой вектор и показывать единицы измерения из `model.units.force` / `model.units.moment`.

# 2 Отдельно реализуемые элементы

## 2.1 Реализовать консоль

- Все функции привязанные к кнопкам должны быть дублированы командами в консоли

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> Требование слишком широкое, если понимать буквально. Нужно реализовать командный слой поверх UI-команд, а не отдельную вторую бизнес-логику. Команда кнопки и команда консоли должны вызывать один и тот же handler.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Минимальный набор команд первой итерации: `help`, `clear`, `fit`, `view.set`, `visibility.set`, `model.validate`, `model.resetSample`, `model.importJson`, `model.exportJson`, `dxf.import`, `settings.show`, `settings.set`, `select.node`, `select.member`, `select.load`.

- Соответственно необходимо реализовать справку по коммандам - help, которая выводит в консоль список команд и их описание

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> `help` должен поддерживать два режима: `help` — список команд; `help commandName` — синтаксис, параметры и пример.

- Так же будут разрабатываться служебные комманды, которые будут доступны только из консоли (например для работы с базой данных)

  <span style="color:#c62828; font-weight:700">КРИТИЧНО:</span> В текущем плане проекта БД не реализуется на данном этапе. Поэтому команды для БД нужно оставить как будущий namespace (`db.*`) без реализации или за feature flag. Нельзя сейчас добавлять реальные DB-команды, если backend/API для этого не согласован.

- Через консоль должны быть доступны настройки проекта

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Настройки проекта должны работать с `model.settings`: `nodeMergeToleranceMm`, `centerModelByXYProjection`, `verticalAxis` только readonly `Z`, а также будущие настройки отображения/панелей. Команды должны валидировать тип и диапазон значений.

## 2.2 Реализовать диалог просмотра каталога профилей и их свойств

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> Нужно различить `каталог профилей проекта` и `глобальный/локальный справочник профилей`. В текущей модели есть `model.profiles` — это профили, используемые или доступные внутри конкретной модели. Отдельный глобальный каталог профилей предусмотрен задачами, но его нужно явно связать с UI.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Диалог профилей должен показывать: `name`, `kind`, параметры сечения, `areaMm2`, `Jx/Iy/Iz`, `Wx/Wy/Wz`, `massKgPerM`, default rotation/offset, цвет профиля, количество элементов, суммарную длину, суммарную массу по профилю.

## 2.3 Реализовать диалог просмотра каталога материалов и их свойств

  <span style="color:#f57c00; font-weight:700">УТОЧНИТЬ:</span> Материалы сейчас есть только в `model.materials`; отдельный справочник материалов не описан как источник данных. На первом этапе можно сделать диалог по материалам текущего проекта, а глобальный справочник материалов вынести в следующую задачу.

  <span style="color:#2e7d32; font-weight:700">ДОБАВИТЬ:</span> Диалог материалов должен показывать: `name`, `elasticModulusMPa`, `shearModulusMPa`, `poissonRatio`, `densityKgPerM3`, `yieldStrengthMPa`, количество элементов, суммарную длину/массу элементов материала.
