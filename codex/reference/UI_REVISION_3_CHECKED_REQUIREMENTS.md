> **Легенда пометок:**
> - <span style="background-color:#fff3cd">желтым</span> — добавлено / уточнено для Codex;
> - <span style="background-color:#f8d7da;color:#842029">красным</span> — противоречие, риск или вопрос на согласование;
> - <span style="background-color:#d1e7dd;color:#0f5132">зеленым</span> — требование согласуется с текущей архитектурой.

<span style="background-color:#fff3cd">**Контекст проверки:** требования ревизии 3 выполняются поверх завершенного блока UI revision 2 (`21.1–21.8`). Перед реализацией нужно добавить отдельный reference-файл, например `codex/reference/UI_REVISION_3_CHECKED_REQUIREMENTS.md`, и новый блок задач после `21.8` с обновлением `IMPLEMENTATION_PLAN.md`, `TASK_SEQUENCE.md`, `TASKS.json` и README/notes при необходимости.</span>

<span style="background-color:#fff3cd">**Общее правило для Codex:** правки дробить на атомарные frontend-задачи: 1) консоль/настройки; 2) DXF import settings + logs; 3) DXF 3D preview + режимы окраски; 4) назначение профилей по цветам; 5) wind dialog polish; 6) docs/checklist. После каждой задачи — `cd frontend && npm run build` и `cd frontend && npm run lint`.</span>

### 1.1 Консоль

- Добавить консольную <span style="background-color:#fff3cd">команду</span> по переключению тем оформления светлая/темная.

  <span style="background-color:#fff3cd">Уточнение: реализовать не как отдельный локальный обработчик, а через существующий `command registry`, чтобы меню, hotkeys и консоль не дублировали бизнес-логику.</span>

  <span style="background-color:#fff3cd">Предлагаемые команды: `theme light`, `theme dark`, `theme toggle`, `theme status`. Состояние темы должно храниться централизованно в app/settings store или аналогичном существующем месте, а `ThemeProvider` должен читать его реактивно.</span>

  <span style="background-color:#f8d7da;color:#842029">Вопрос/риск: текущая UI revision 2 закрепила black & white / grayscale-ориентированную тему. Нужно подтвердить, что `light/dark` — это два режима той же grayscale-эстетики, а не возврат к цветной MUI-теме.</span>

## 1.2 импорт DXF

- Настройки импорта из интерфейса убираем, настройки (и их изменение) должны быть доступны через консольные <span style="background-color:#fff3cd">команды</span>.

  <span style="background-color:#d1e7dd;color:#0f5132">Согласуется с текущей архитектурой: настройки DXF уже выделены как `DxfImportSettings` (`toleranceMm`, `centerOnXY`, `force2DToXY`), поэтому их можно перенести из диалога в console/settings-команды без изменения онтологии модели.</span>

  <span style="background-color:#fff3cd">Уточнение: после удаления настроек из UI в диалоге нужно оставить read-only summary текущих настроек и кнопку/подсказку “изменить через консоль”, иначе пользователь не поймет, с какими параметрами выполняется импорт.</span>

  <span style="background-color:#fff3cd">Предлагаемые команды: `dxf.settings`, `dxf.tolerance 1`, `dxf.center on|off`, `dxf.force2d on|off`, `dxf.settings reset`.</span>

- Кнопку "Выбрать DXF файл" перенести в низ к остальным кнопкам (все кнопки реализуем блоком)

  <span style="background-color:#fff3cd">Уточнение: блок кнопок внизу должен включать: выбрать/заменить файл, открыть логи, отмена, импорт. Для выбора файла оставить скрытый `input type=file`, а видимую кнопку сделать через MUI `IconButton`/`ButtonBase` с tooltip.</span>

- Все кнопки - icon button

  <span style="background-color:#fff3cd">Уточнение: для icon button обязательно добавить `Tooltip` и `aria-label`; для критичных действий (`Импорт`, `Отмена`) можно использовать компактный icon button с текстовой подписью в tooltip, чтобы не потерять доступность и понятность.</span>

- Вместо списка с предупреждениями добавить 3d сцену просмотра импорта, в которой применить цветовую дифференциацию импорта (зеленый - норм, желтый - предупреждение, красный - ошибка)

  <span style="background-color:#fff3cd">Уточнение: 3D preview должен строиться до применения импорта к основной модели и не должен менять `modelStore` до нажатия “Импорт”. Желательно использовать общий renderer/компоненты viewport, но на временной preview-модели.</span>

  <span style="background-color:#f8d7da;color:#842029">Риск/добавить в задачу: текущий `DxfImportPreview` хранит warnings/errors как общие строки, а не как диагностику по конкретным стержням/узлам. Для окраски отдельных элементов в красный/желтый нужно расширить preview-данные: добавить per-member/per-node diagnostics или хотя бы `memberImportStatuses` с привязкой к `memberId`/DXF handle.</span>

  <span style="background-color:#fff3cd">Минимально допустимая реализация: если диагностика пока только общая, окрашивать всю preview-модель по общему статусу нельзя считать достаточным решением. Нужно хотя бы подсветить: нулевые/ошибочные линии, висячие стержни, группы с неназначенным профилем, ignored entities в логах.</span>

- Так же нужен переключатель просмотра между режимом отображения ошибок и цветовой дифференцировкой профилей

  <span style="background-color:#fff3cd">Уточнение: режимы назвать, например: `Диагностика` и `Профили`. В режиме `Диагностика`: зеленый/желтый/красный по статусу импорта. В режиме `Профили`: цвет элемента соответствует DXF color/layer group или назначенному профилю.</span>

  <span style="background-color:#f8d7da;color:#842029">Вопрос: “цветовая дифференцировка профилей” — окрашивать по исходному цвету DXF, по назначенному профилю из каталога или по типу профиля (уголок/полоса/квадрат)? Сейчас ниже указано “цвет -> профиль”, поэтому я трактую как исходный цвет DXF / layer group → выбранный профиль.</span>

- Диалог назначения профиля нужно реализовать по принципу: отображение цвета -> профиль, добавить выбор типа профиля (уголок / полоса / квадрат)

  <span style="background-color:#d1e7dd;color:#0f5132">Согласуется с текущей логикой DXF: цвет/слой уже используется для группировки стержней и назначения профилей.</span>

  <span style="background-color:#fff3cd">Уточнение: тип профиля должен быть фильтром каталога, а не новым полем модели. Порядок выбора: строка группы цвета → select “тип профиля” → select “профиль” с фильтрацией по типу. При смене типа, если текущий профиль не подходит, сбрасывать выбранный профиль.</span>

  <span style="background-color:#f8d7da;color:#842029">Вопрос: нужны только три типа `уголок / полоса / квадрат`, или нужно сохранить все фактические `profile.kind` из каталога, а эти три вывести первыми? Если в каталоге уже есть иные типы, жесткое ограничение может скрыть существующие профили.</span>

- Предупреждения импорта и логи вынести в отдельное открывающееся окно (открываем по кнопке логи)

  <span style="background-color:#fff3cd">Уточнение: окно логов должно показывать summary импорта, warnings, errors, ignored entities count, параметры импорта, color/layer groups, source metadata. Это не должно заменять глобальный MUI Alert/Snackbar: snackbar остается для краткого результата операции, подробности — в modal/drawer логов.</span>

- служебную информацию убрать, перенести ее в логи (просмотр в окне логирования как summary)

  <span style="background-color:#d1e7dd;color:#0f5132">Согласуется с текущим направлением UI revision 2: пользовательский интерфейс очищается, служебные данные доступны через консоль/логи/debug-команды.</span>

  <span style="background-color:#fff3cd">Добавить: в основном диалоге DXF оставить только имя файла, 3D preview, режим окраски, compact-status и кнопки. Все метрики (`linesCount`, `nodesCount`, `membersCount`, `ignoredEntitiesCount`, `zRange`, `mergedNodesCount`, `danglingMembersCount`) перенести в окно логов.</span>

## 1.3 Диалог ввода параметров ветра

- Заголовок поменять на Ветер

  <span style="background-color:#d1e7dd;color:#0f5132">Согласуется: текущий заголовок “Ручной ветер” можно заменить без изменения модели.</span>

- Поле загружение - увеличить вертикальный отступ от заголовка, отступы в форме должны быть одинаковые

  <span style="background-color:#fff3cd">Уточнение: выровнять spacing через единый `Stack spacing`, без точечных margin-исключений.</span>

- Поле Z - убрать, Z считаем равным нулю по умолчанию

  <span style="background-color:#fff3cd">Уточнение: при сохранении всегда передавать `direction.z = 0`; для существующих моделей с ненулевым `z` при открытии диалога показывать warning в форме/логах и при сохранении явно нормализовать к нулю.</span>

  <span style="background-color:#f8d7da;color:#842029">Риск: ранее модель ветра была заложена как 3D-вектор направления. Удаление поля `Z` из UI допустимо только как UI-ограничение текущей итерации; тип модели и API пересчета ГСК/ЛСК менять нельзя.</span>

- Форму сделать уже на треть

  <span style="background-color:#fff3cd">Уточнение: задать `maxWidth` диалога/контента примерно на 30–35% меньше текущего, но сохранить адаптивность для узких экранов (`xs: 100%`).</span>

- запретить ввод отрицательного значения для ветрового давления

  <span style="background-color:#fff3cd">Уточнение: запретить на уровне input (`min=0`) и на уровне submit validation. При ручном вводе отрицательного числа показывать validation error, а не молча игнорировать.</span>

  <span style="background-color:#fff3cd">Добавить: определить, допускается ли `0`. Логично оставить `0` как “ветер отключен/давление отсутствует”, но отрицательные значения запретить.</span>

- поля ввода X, Y, номинальное давление - NumberField, для X, Y инкремент - 1, для номинальное давление инкремент - 10

  <span style="background-color:#fff3cd">Уточнение: если собственного `NumberField` в shared/ui нет — создать общий компонент, чтобы не плодить разные настройки `TextField type="number"`. Для `nominalPressureKPa` в модели единица — kPa, а инкремент `10` выглядит крупным для kPa; нужно подтвердить единицы.</span>

  <span style="background-color:#f8d7da;color:#842029">Вопрос: номинальное давление сейчас хранится как `nominalPressureKPa`. Инкремент `10` означает 10 кПа, что для ветра очень много. Возможно, подразумевались Па или кгс/м²? Если точно кПа — оставляю требование как задано, но это стоит подтвердить.</span>

- кнопки - icon button

  <span style="background-color:#fff3cd">Уточнение: как и для DXF, добавить `Tooltip` и `aria-label`; для submit/cancel желательно сохранить ясные tooltip-подписи “Сохранить” и “Отмена”.</span>

- В выпадающем списке загружений писать только наименования загружений

  <span style="background-color:#d1e7dd;color:#0f5132">Согласуется: сейчас в списке отображается `name (id)`, нужно убрать `id` из пользовательского UI.</span>

  <span style="background-color:#fff3cd">Добавить: если имена загружений не уникальны, в UI можно показывать только имя, но внутренне `value` должен оставаться `loadCase.id`; при дублирующихся именах стоит добавить tooltip/id только в логах или debug, а не в самом списке.</span>
