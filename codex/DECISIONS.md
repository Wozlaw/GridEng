# Зафиксированные решения по итерации после 9.2

1. Распределенные нагрузки на стержень реализуем сразу, потому что это часто используемый расчетный сценарий.
2. Все нагрузки задаются только в ГСК. Пересчет в ЛСК будет выполняться через API в будущей итерации.
3. Для распределенной нагрузки используем общий вариант `linear` с `qStart` и `qEnd`. Частные случаи:
   - равномерная: `qStart === qEnd`;
   - линейно возрастающая: `qEnd > qStart`;
   - линейно убывающая: `qEnd < qStart`.
4. Функциональную нагрузку сейчас не реализуем, но модель должна быть расширяемой так, чтобы позже можно было добавить distribution по функции без ломки структуры.
5. Поле `comment` добавляем к пользовательским сущностям: `Node`, `Member`, `Restraint`, `Load`, `LoadCase`, `Profile`, `Material`, `WindLoadDefinition`.
6. Материал остается свойством каждого отдельного элемента (`member.materialId`). Не переносить материал внутрь профиля. В будущем можно добавить `defaultMaterialId`, но не в этой задаче.
7. Переключение языка делаем через консоль, меню не засоряем. Архитектура локализации должна поддерживать `ru` и `en`.
8. Все ошибки, служебные сообщения, предупреждения и сообщения о не реализованном функционале показываем через единый MUI Alert/Snackbar снизу по центру.
9. Заглушки будущего функционала показывают MUI Alert, не нативный `window.alert`.
10. `Каталог конструкций` — будущая backend-сущность: атлас типовых опор с типовыми нагрузками по расчетным моделям. Сейчас только placeholder через Alert.
11. Ветровая нагрузка в этой итерации — только ручной ввод направления и номинального давления. Нормативный калькулятор не делать.
12. БД не реализовывать. Консольный namespace `db.*` можно зарезервировать только как placeholder/help, реальные команды не добавлять.
